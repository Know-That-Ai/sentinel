# Sentinel

Watches GitHub PRs for scanner activity (BugBot, CodeQL, CI failures) and
routes each event straight into the Claude Code session that's working on
that PR. Scanner comments get typed as prompts into the right iTerm / Terminal
tab, Claude starts addressing them immediately, you keep shipping.

```
  GitHub webhook
        │
        ▼
  smee.io channel   (public relay, dev-friendly)
        │
        ▼
  smee-client  →  localhost:3847/webhook      (daemon, started by launchd)
        │
        ▼
  match (repo, PR#)  →  linked session (repo, PR#, tty)
        │
        ▼
  AppleScript / tmux → write into the exact iTerm/Terminal tab running Claude
```

macOS only. iTerm2, Terminal.app, and tmux are all first-class injection targets.

---

## Prerequisites

- macOS (osascript + launchd are required)
- Node **22+**, **pnpm**, **Rust toolchain** (for the TUI — installed automatically if you have `rustup`)
- A terminal emulator with AppleScript support — **iTerm2** or **Terminal.app** — OR tmux (works inside any terminal)
- A GitHub PAT with access to the repos you want to monitor:
  - **Classic**: `repo` + `admin:repo_hook`
  - **Fine-grained** (per-repo): `Metadata: Read`, `Pull requests: Read & Write`, `Contents: Read`, `Webhooks: Read & Write`
- A [smee.io](https://smee.io) channel URL (auto-generated by `sentinel webhooks --rotate-smee` — see below)

## Quick start

```bash
git clone https://github.com/Know-That-Ai/sentinel
cd sentinel
./scripts/install.sh
```

The installer:

1. `pnpm install` + `pnpm build` + `pnpm build:tui` (Rust)
2. Creates `~/.sentinel/` (holds the SQLite DB and logs)
3. Copies `.env.example` → `.env` (edit this next)
4. Symlinks the `sentinel` CLI into your `$PATH`
5. Adds a Claude Code `PostToolUse` hook at `~/.claude/settings.json` so sessions auto-link when Claude's output mentions a PR URL
6. Registers a launchd agent (`com.sentinel.daemon`) so the daemon + menubar UI + smee proxy start on login

After install, edit `/Users/<you>/@kya-os/sentinel/.env`:

| Variable | What to put | Required |
|---|---|---|
| `GITHUB_PAT` | your token | ✅ |
| `GITHUB_ORG` | e.g. `Know-That-Ai` | ✅ |
| `GITHUB_USERNAME` | your GitHub handle | ✅ |
| `WEBHOOK_SECRET` | any random 32+ char string | ✅ |
| `SCANNER_BOT_LOGINS` | comma-sep bot accounts to react to, e.g. `cursor-bugbot[bot],github-advanced-security[bot]` | ✅ |
| `REPO_PATHS` | JSON map, e.g. `{"Know-That-Ai/sentinel":"/Users/me/@kya-os/sentinel"}` | ✅ |
| `SMEE_URL` | leave unset — `sentinel webhooks --rotate-smee` generates one | ⚠️ |
| `SENTINEL_USER_LABEL` | your name, for multi-user environments | optional |
| `SENTINEL_AUTO_SUBMIT` | `true` (default — focus the Claude tab and press Enter) / `false` (type only) | optional |
| `AUTO_DISPATCH_BUGBOT/CODEQL/CI` | `true` to spawn an agent for *unlinked* PRs; `false` (default) to just notify | optional |
| `SENTINEL_DEFAULT_TMUX_PANE` | only set if you run Claude inside a named tmux pane | optional |

Restart the daemon so it picks up your `.env`:

```bash
launchctl kickstart -k gui/$(id -u)/com.sentinel.daemon
```

## Wire up GitHub webhooks

One command, from anywhere:

```bash
sentinel webhooks --rotate-smee
```

What that does:

1. `POST https://smee.io/new` — generates a fresh smee channel
2. Writes `SMEE_URL=https://smee.io/<channel>` into your `.env`
3. Kicks the daemon so the built-in smee-client reconnects to the new URL
4. For every active repo in your `GITHUB_ORG`, creates (or updates) a GitHub webhook pointing at that URL

If you already have a `SMEE_URL` set and just want to (re)register hooks:

```bash
sentinel webhooks
```

Idempotent — re-running skips repos that already have the correct webhook. Rotating (`--rotate-smee`) updates existing smee-pointing hooks in place instead of leaving orphans.

Per-repo errors (e.g. PAT lacks webhook permission on one repo) don't abort the run; they're listed at the end.

## Link a Claude session to a PR

This is the core user action. Sentinel needs to know *which iTerm tab* is working on *which PR*. One of three ways:

### 1. Automatic via the Claude Code hook (recommended)

The installer added a `PostToolUse` hook to `~/.claude/settings.json` that runs `sentinel link` any time Claude's bash output contains a `github.com/.../pull/N` URL. Just start working — the first time Claude calls `gh pr view`, the link is created.

### 2. From inside Claude — `! sentinel link`

Type `!` at Claude's prompt followed by the command. Claude runs it as a shell command in the same tab, ~200ms, no quit/resume. This captures the tab's tty via the parent-walking logic in `cmdLink`, so the link is correctly scoped.

### 3. Before launching Claude

From your repo's directory:

```bash
sentinel link       # auto-detects the open PR for the current branch
sentinel link --pr 42 --repo Know-That-Ai/agent-shield   # explicit
```

Then start `claude` in the same tab.

### Git worktrees

Worktrees are fully supported. Each worktree has its own `HEAD`, so you can link a different PR from each one. Run `sentinel link` from inside the worktree directory — never from the base repo.

### Verify

```bash
sentinel tui        # Sessions tab — one row per linked PR
sentinel status     # text version
```

Each row should show a real `tty` (e.g. `/dev/ttys016`). A `-` means the capture couldn't resolve the tty; events for that link will only write the inbox file (see below) without typing.

## What happens when a scanner fires

1. GitHub sends the webhook event to your smee channel.
2. smee-client forwards it to `localhost:3847/webhook`.
3. The handler classifies the event. Sender must match `SCANNER_BOT_LOGINS` for scanner comments, or be a `check_run` for BugBot/CodeQL/CI.
4. Event is written to SQLite (`~/.sentinel/sentinel.db`).
5. If there's a live linked session for `(repo, PR#)`:
   - A short trigger line is typed into the linked tab:
     > `Sentinel: 3 bugbot issues on PR #1830 — read .sentinel/inbox/1830.md and address per the instructions in that file.`
   - The **full** multi-line prompt (with instructions, push rules, each finding broken out) is written to `.sentinel/inbox/<pr>.md` inside the repo/worktree.
   - Claude types/submits the trigger → reads the file → starts working.
6. If there's **no** linked session, a menu-bar notification fires instead. The `AUTO_DISPATCH_*` flags decide whether to also auto-spawn an agent — by default they just notify.

The inbox file is the durable artifact. If your iTerm tab is closed or Sentinel can't inject for any reason, the file still lands on disk and Claude can `cat` it.

## Day-to-day commands

```bash
# link management (from the iTerm tab where Claude runs)
sentinel link                         # link current branch's open PR
sentinel link --pr 42 --repo o/r      # explicit
sentinel unlink                       # detach this repo's link

# observe
sentinel tui                          # full dashboard — dashboard / sessions / events / config
sentinel status                       # terse text version

# service control
sentinel start | stop | restart       # launchd daemon
sentinel logs                         # tail ~/.sentinel/*.log

# ops
sentinel webhooks                     # (re)register webhooks from .env
sentinel webhooks --rotate-smee       # fresh smee URL + update all hooks in place
sentinel flush --pr 42                # re-fetch and re-dispatch last scanner runs for a PR
sentinel test-webhook --pr 42 --type bugbot   # dry-run: fake 3 BugBot findings, full dispatch path
sentinel test-webhook --pr 42 --type codeql
sentinel test-webhook --pr 42 --type ci
sentinel test-webhook --pr 42 --type success  # mark PR green, fire the green-state path
sentinel uninstall                    # remove the Claude Code PostToolUse hook
```

`sentinel test-webhook` must run from inside the target repo (it uses git remote to determine `owner/repo`). Run it from a *different* iTerm window than the linked one to see the cross-window routing work.

## The TUI

```bash
sentinel tui
```

Four tabs:

| Tab | Shows |
|---|---|
| Dashboard | Daemon health, port, smee URL, `at-a-glance` counts (sessions, unreviewed events, scanner events, bots watched) |
| Sessions | Every linked PR with repo · PR# · agent · pid · tty · linked-at. Details pane shows repo path and tmux pane. |
| Events | Unreviewed scanner events. Detail pane resolves which linked session the event will dispatch into, or "no linked session (falls back to notification)". |
| Config | `SCANNER_BOT_LOGINS` (who we react to), `REPO_PATHS`, unlinked-PR fallback flags, injection mode (auto-submit toggle), webhook endpoints |

Key bindings:
- `Tab` / `1-4` — switch tab
- `j`/`k` — navigate list
- `r` — refresh (on Events tab: mark selected reviewed)
- `d` — on Events tab: dispatch selected event into its linked session
- `o` — open selected PR in the browser
- `q` — quit

## Troubleshooting

**`sentinel link` prints `tty -` (null).** You ran it somewhere that has no controlling terminal — usually Claude Code's `!` path on older builds. Make sure you're on `main` (commit `fa38af3` or later) which walks up the process tree to find a tty.

**`sentinel tui` Sessions tab is empty but `sentinel link` succeeded.** You're pre-commit `d274aff` (different DBs per cwd). Update, kill stray `sentinel.db*` files outside `~/.sentinel/`, and relink.

**`No open PR found for the current branch`.** Either you're in a worktree / base repo with the wrong branch checked out, or the PR was closed/merged, or (for fine-grained PATs) the token doesn't have access to that repo. Use explicit flags:
```bash
sentinel link --pr 1830 --repo Know-That-Ai/agent-shield
```

**`tmux send-keys` error when no tmux is running.** You have `SENTINEL_DEFAULT_TMUX_PANE` set in `.env` but aren't actually running tmux. Comment out or delete that line, then `sentinel unlink && sentinel link` to refresh the stored pane.

**Scanner comment arrives but nothing types into Claude.** A few checks in order:
1. `sentinel tui` → Events — is the event actually in Sentinel? If not, webhook/smee issue.
2. Same tab — is the event's "will dispatch into" pointing at your linked session? If it says "no linked session", the link's `repo` field probably doesn't match (different case, different org, linked from wrong cwd).
3. Does the linked session have a real `tty`? If `-`, relink from the Claude tab.
4. Terminal emulator — iTerm2, Terminal.app, or tmux. Ghostty / Kitty / Alacritty / WezTerm / Warp / Hyper have no AppleScript surface; inbox file is written but no typing happens. Run Claude inside tmux and set `SENTINEL_DEFAULT_TMUX_PANE` to get universal delivery.

**Focus keeps snapping to the Claude tab.** That's intentional so a real Enter keystroke registers (iTerm's `write text` wraps in bracketed paste; an embedded CR there isn't treated as submit). If you'd rather review before submitting, set `SENTINEL_AUTO_SUBMIT=false` in `.env` and restart the daemon — Sentinel will type the trigger line but leave Enter to you.

**404 from GitHub on `sentinel link` / `webhooks`.** PAT scope. Fine-grained PATs are per-repo; GitHub returns 404 (not 403) for repos outside the allowlist. Either extend the PAT's "Only select repositories" list or switch to a classic PAT with `repo` + `admin:repo_hook`.

## Uninstall

```bash
sentinel uninstall                                           # remove Claude Code hook
launchctl unload ~/Library/LaunchAgents/com.sentinel.daemon.plist
rm -rf ~/.sentinel ~/Library/LaunchAgents/com.sentinel.daemon.plist
rm ~/.local/bin/sentinel
```

And delete the repo directory if you're done with it.

## Layout

```
src/
  cli.ts                  # the `sentinel` command
  main.ts                 # daemon entry — HTTP server + poller + PID watcher
  server/routes.ts        # HTTP: /health, /state/*
  agents/
    index.ts              # dispatch flow (link lookup → inject vs notify)
    inject.ts             # tmux / iTerm2 / Terminal.app injection
    prompts.ts            # prompt templates
  github/
    events.ts, repos.ts, octokit.ts, poller.ts
  db/
    index.ts, schema.ts, queries.ts
  menubar/                # Electron menu-bar UI
tui/                      # Rust / Ratatui dashboard
scripts/
  install.sh, install-hook.ts, setup-webhook.ts, smee-dev.ts
```

The daemon, the Electron UI, the Rust TUI, and the CLI all talk to the same
SQLite DB at `~/.sentinel/sentinel.db`. The TUI and menubar UI additionally
talk HTTP to the daemon on port 3847 for write actions (mark-reviewed,
dispatch).
